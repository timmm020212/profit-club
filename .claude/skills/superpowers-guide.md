# Справочник Superpowers

Superpowers — фреймворк структурированной разработки. Три фазы: Дизайн -> Планирование -> Исполнение.

## Команды

### Фаза 1: Дизайн (что делаем?)

**`/brainstorming`** — Мозговой штурм перед любой творческой работой
- Когда: перед созданием фичи, компонента, изменением поведения
- Что делает: исследует намерения пользователя, требования, ограничения
- Результат: чёткое понимание ЧТО нужно сделать и ЗАЧЕМ
- Пример: "хочу добавить аналитику" → обсуждение: какие метрики, для кого, какой UI

### Фаза 2: Планирование (как делаем?)

**`/writing-plans`** — Написать план реализации
- Когда: есть спецификация/требования, нужен пошаговый план
- Что делает: создаёт детальный план с маленькими задачами (2-5 мин каждая)
- Результат: файл плана с чёткими шагами, файлами, критериями готовности
- Пример: план миграции БД → 12 шагов с конкретными файлами и командами

### Фаза 3: Исполнение (делаем!)

**`/executing-plans`** — Выполнить написанный план
- Когда: есть готовый план, нужно его реализовать
- Что делает: выполняет план пошагово с контрольными точками ревью
- Результат: реализованный код по плану

**`/subagent-driven-development`** — Параллельное выполнение через подагентов
- Когда: в плане есть независимые задачи которые можно делать параллельно
- Что делает: запускает несколько агентов одновременно на независимых задачах
- Результат: быстрая реализация нескольких фич одновременно

**`/dispatching-parallel-agents`** — Распределить 2+ независимых задач
- Когда: есть несколько несвязанных задач без общего состояния
- Что делает: параллельно запускает агентов на каждую задачу
- Результат: все задачи выполнены параллельно

### Качество

**`/test-driven-development`** — TDD: сначала тесты, потом код
- Когда: перед написанием кода фичи или фикса бага
- Что делает: пишет тесты → запускает (красные) → пишет код → запускает (зелёные)
- Результат: код покрытый тестами

**`/systematic-debugging`** — Системный дебаг
- Когда: баг, падение теста, неожиданное поведение
- Что делает: собирает факты → формулирует гипотезы → проверяет каждую → фиксит
- Результат: найденная и исправленная причина бага

**`/verification-before-completion`** — Верификация перед завершением
- Когда: перед тем как сказать "готово", перед коммитом или PR
- Что делает: запускает проверки, подтверждает что всё работает
- Результат: доказательства что код работает (не просто "должно работать")

### Ревью

**`/requesting-code-review`** — Запросить ревью кода
- Когда: после завершения задачи, перед мержем
- Что делает: проверяет код на соответствие требованиям и стандартам
- Результат: список замечаний или подтверждение качества

**`/receiving-code-review`** — Получить и обработать ревью
- Когда: получил фидбек на код, перед тем как слепо применять
- Что делает: анализирует замечания, проверяет их техническую корректность
- Результат: обоснованные изменения (не слепое согласие)

### Git и завершение

**`/using-git-worktrees`** — Работа в изолированном worktree
- Когда: начало фичи, нужна изоляция от основной ветки
- Что делает: создаёт git worktree, переключается туда
- Результат: изолированное рабочее пространство

**`/finishing-a-development-branch`** — Завершение ветки разработки
- Когда: фича готова, тесты проходят, нужно интегрировать
- Что делает: предлагает варианты: merge, PR, cleanup
- Результат: код интегрирован в основную ветку

### Утилиты

**`/writing-skills`** — Создать или отредактировать скилл
- Когда: нужен новый скилл или правка существующего
- Что делает: создаёт .md файл скилла с правильной структурой

**`/using-superpowers`** — Стартовый скилл
- Когда: начало разговора, помогает определить какой скилл использовать

## Рекомендуемый порядок работы

```
/brainstorming        → понять ЧТО делаем
/writing-plans        → спланировать КАК
/executing-plans      → реализовать
/verification-before-completion → проверить
/requesting-code-review → ревью
/finishing-a-development-branch → завершить
```
